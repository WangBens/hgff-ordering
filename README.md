# 🚀 凰貴妃茶堂 - LINE 線上點餐系統開發全紀錄
[🇹🇼 繁體中文](README.md) | [🇺🇸 English](README_en.md)

---
> **開發日誌與心得**
> 這是一段從無到有、充滿挑戰的開發旅程。在專案初期，我對整體架構與功能流程並不清楚。透過釐清需求，我將系統拆解為「菜單顯示」、「加入購物車」、「訂單管理」與「結帳計算」等核心模組。在實作過程中，經歷了資料格式不一、按鈕失效、購物車無法正確累加、畫面未同步更新，甚至型別錯誤導致金額計算異常等各種問題。透過不斷地重構邏輯、分離功能與優化 UI，我最終完成了一套具備商業水準的系統。這段過程讓我學會如何獨立拆解問題、除錯與優化邏輯，大幅提升了程式開發的能力。

---

## 一、 系統架構與資料庫設計 (後端)

本系統確立以 Google Sheets 作為輕量級資料庫，搭配 Google Apps Script (GAS) 提供 API 服務，並完美整合 LINE LIFF 達成無縫登入的架構。

* **自動化建表：** 撰寫了 `ensureSheetsExist` 函式，讓系統初次運行時，自動建立包含系統設定、管理員名單、會員餘額、交易明細、菜單、客製化選項、訂單紀錄與促銷活動等資料表。
* **強大的折扣引擎：** 在後端開發了複雜的計價模型 (`calculateBackendTotal`)，支援「全館/系列/單品買 X 送 Y」以及「指定組合套餐價」。系統會將購物車攤平 (Flatten) 並由高價至低價優先折抵，確保對消費者最有利。

---

## 二、 顧客端介面開發 (INDEX.html)

為了讓顧客有最好的手機點餐體驗，透過 LINE LIFF 打造了專屬的顧客端網頁。

* **會員與錢包系統：** 顧客開啟網頁即自動帶入 LINE 資訊，支援一鍵加入會員。系統能即時顯示會員專屬餘額，並提供完整的歷史交易與訂單明細查詢。
* **動態菜單與跑馬燈：** 讀取後台設定，自動產生分類菜單。若有啟用的促銷活動，頂部會自動顯示活動跑馬燈，並獨立拉出「今日優惠組合」專區。
* **高度客製化購物車：** 顧客可以自由選擇杯型 (L/XL)、甜度、冰塊，並且支援「複選加料」。
* **無縫結帳體驗：** 支援「現場結帳」與「餘額扣款」兩種模式。結帳畫面上能即時、精準地顯示系統自動套用的活動折抵明細與最終總額。

---

## 三、 老闆管理端開發 (BOSS.html)

後台系統是營運的核心，設計了直覺的圖形化介面供管理員操作。

* **門市狀態與即時看板：** 可一鍵切換「營業中/休息中」狀態。訂單看板每 15 秒自動刷新，提供「接單、完成、已領」的訂單流程狀態按鈕。
* **菜單與活動管理：** 支援快速上架商品與雙杯型價格設定。強大的活動設定介面可針對單品或組合設定優惠，並支援隨時下架商品或停用活動（系統具備防呆機制，避免誤下架綁定活動中的商品）。
* **數位錢包管理：** 老闆可以直接在後台為指定顧客手動執行「儲值」或「扣款」，並填寫帳務備註，所有資金流動皆會寫入交易明細中供稽核。

## 四、 系統安全機制 (Security Mechanisms)

為了將系統提升至商業可用層級，本系統在前後端實作了多層防護與安全機制：

* **LINE ID Token 身分驗證：** 系統不依賴前端容易被竄改的 `userId` 進行 API 請求，而是強制前端發送 `idToken`，由後端向 LINE 伺服器進行二次驗證，確保操作者身分真實性。
* **防重複提交 (Request ID Cache)：** 在結帳或修改餘額等寫入動作中，前端會自帶隨機的 `requestId`，後端利用 `CacheService` 將其鎖定 60 秒。即便顧客在網路卡頓時瘋狂連點，也只會產生一筆訂單。
* **流量限制 (Rate Limit)：** 針對每位使用者的 API 呼叫頻率設定上限（寫入操作每分鐘最高 60 次，讀取 180 次），有效防禦惡意腳本灌水或 DDoS 攻擊。
* **並發鎖定 (LockService)：** 在處理「餘額扣款」與「接單退款」的程式區塊加入 `LockService.getScriptLock().waitLock(10000)`，防止並發寫入 (Race Condition) 導致餘額被扣成負數或訂單狀態異常。
* **嚴格的權限隔離：** 老闆後台專用的 API 指令會強制核對發送者是否在「管理員名單」中，無權限者即使取得 API 網址也無法操作。

## 五、 免費版效能預估與系統限制 (Capacity)

這套系統完全利用免費資源架設，以下為乘載量的安全預估：

* **前端 GitHub Pages (網頁代管)：**
  * GitHub Pages 擁有極大的頻寬（每月約 100GB 容量），由於網頁只是純靜態的 HTML/JS，支援 **數千至數萬人同時開啟網頁** 毫無壓力。
* **後端 Google Apps Script (API 伺服器)：**
  * GAS 的免費配額為每日約 20,000 次 URL Fetch（用於驗證 LINE Token），以及每日總執行時間 90 分鐘。
  * **同時上線預估：** 系統約可穩定支撐 **30~50 人「在同一秒鐘內」同時送出訂單或頻繁刷新畫面**。對於一般中小型飲料門市或校園商圈，這個併發量絕對綽綽有餘。若瞬間併發過高，部分請求可能會稍慢或需重試。
* **資料庫 Google Sheets：**
  * **容量上限：** Google 試算表單一檔案最高支援 **1,000 萬個儲存格**。以本系統一筆訂單佔用約 10 個儲存格（含明細欄位）來計算，累積 **幾十萬筆訂單** 也不會達到上限。
  * *維護建議：* 為了保持系統在讀取歷史紀錄時的極致流暢，建議每隔半年或一年，手動將舊的「訂單紀錄」與「交易明細」複製到另一個試算表備份並清空原始工作表。

## 六、 關鍵技術挑戰與完美解法

**⚠️ 挑戰 1：後台訂單數量的「假顯示」問題**
* **狀況：** 後端計價為了精準計算折扣，將所有同款飲料拆解成「單杯」陣列。導致老闆看後台時，買 5 杯紅茶變成 5 行獨立紀錄。
* **解法：** 在 `BOSS.html` 導入 `groupedItems` 邏輯，重新聚合相同「杯型、品名、甜度、冰塊、加料」的品項並加總 `qty`，讓明細合併顯示為「[L] 紅茶 x5」。

**⚠️ 挑戰 2：前後端優惠計算不一致的致命傷**
* **狀況：** 顧客在購物車看到的金額，與送出訂單後端的實際扣款金額出現落差。前端原本無法完美處理「組合價」與「買送」的交錯折扣。
* **解法：** 執行邏輯百分之百同步。將後端嚴謹的攤平排序計價邏輯 (`calculateBackendTotal`)，完整無誤差地移植到前端的 `calculateCart()` 中。

**⚠️ 挑戰 3：API 延遲導致畫面卡死 (UX 痛點)**
* **狀況：** 網路不穩或 GAS 伺服器回應慢時，`fetch()` 陷入無限等待，畫面卡在「處理中...」。
* **解法：** 在前後端的 `callGAS()` 封裝中，全面導入 `AbortController` 實作 15 秒 Timeout (超時) 機制。連線過久自動中斷並提示「伺服器回應逾時，請稍後再試」。

**⚠️ 挑戰 4：殘留程式碼造成的語法錯誤**
* **狀況：** 反覆修正前端計價邏輯時，將一段迴圈遺漏在主函數外部，引發 `SyntaxError` 導致白畫面。
* **解法：** 重新梳理程式碼區塊，清除殘留的孤立程式碼，重構作用域 (Scope)，恢復正常渲染。

## 七、 系統建置與部署教學 (Step-by-Step)

若要在新環境架設本系統，請依序執行以下步驟。

### 📌 自行架設必改清單
在開始之前，請確認你有以下三個檔案，並且在部署過程中必須修改這三個地方的 ID：
1. `GS` (後端)：修改 `LINE_CLIENT_ID`
2. `INDEX.html` (顧客端)：修改 `CUSTOMER_LIFF_ID` 與 `GAS_API_URL`
3. `BOSS.html` (老闆端)：修改 `BOSS_LIFF_ID` 與 `GAS_API_URL`

### 第一步：申請 LINE Developers 與 LIFF ID (準備前端入口)
1. 前往 **[LINE Developers Console](https://developers.line.biz/)** 並登入 LINE 帳號。
2. 建立 **Provider**，並在該 Provider 下建立一個 **LINE Login** 頻道的 Channel。
3. 進入 Channel 設定頁面，記下 **Channel ID** (這就是後端的 `LINE_CLIENT_ID`)。
4. 切換到 **LIFF 頁籤**，點擊「Add」新增兩個 LIFF App：
   * **顧客端 LIFF**：Size 設為 `Full` 或 `Tall`，Scope 勾選 `profile` 與 `openid`。
   * **老闆端 LIFF**：Size 設為 `Full`，Scope 勾選 `profile` 與 `openid`。
5. 記下這兩個 LIFF App 產生的 **LIFF ID** 待用。

### 第二步：建立 Google Sheets 與 Apps Script (建置資料庫與後端 API)
1. 建立一個全新的 **Google 試算表**，點擊上方選單 **擴充功能 > Apps Script**。
2. 貼上 `GS` (後端程式碼)，並將最上方的 `LINE_CLIENT_ID` 換成第一步的 Channel ID。
3. **🔑 關鍵步驟：觸發 Google 授權**
   * 在程式碼編輯器上方的工作列，將執行的函式從下拉選單改為 `ensureSheetsExist`。
   * 點擊「執行」按鈕。
   * 系統會跳出「需要授權」視窗 $\rightarrow$ 點擊「審查權限」 $\rightarrow$ 選擇你的 Google 帳號 $\rightarrow$ 點擊「進階」 $\rightarrow$ 點擊「前往（不安全）」 $\rightarrow$ 點擊「允許」。
4. 授權完成後，點擊右上角的 **部署 > 新增部署作業**。
5. 類型選擇 **網頁應用程式 (Web App)**，執行身分設為 **我**，誰可以存取設為 **所有人 (Anyone)**。
6. 部署完成後，複製 **網頁應用程式網址 (Web App URL)**，此為 `GAS_API_URL`。

---

### 💡 關鍵步驟：強制開啟 Google 服務權限 (Auth Trigger)
為了讓你的 Google Apps Script 順利連線外部 API (LINE) 並存取你的試算表，你在**部署前**必須先手動觸發一次權限請求。請將以下程式碼貼入你的 Apps Script 編輯器並執行一次：

```javascript```
`function forceAuth() {
  // 這段程式碼呼叫了三個不同的服務，會強制引導 Google 跳出授權請求視窗
  UrlFetchApp.fetch("[https://api.line.me/](https://api.line.me/)");
  SpreadsheetApp.getActiveSpreadsheet().getName();
  LockService.getScriptLock();
}`

### 第三步：綁定金鑰與部署前端網頁 (INDEX.html 與 BOSS.html)
1. 編輯 **INDEX.html**，將 `CUSTOMER_LIFF_ID` 與 `GAS_API_URL` 替換為你的數值。
2. 編輯 **BOSS.html**，將 `BOSS_LIFF_ID` 與 `GAS_API_URL` 替換為你的數值。
3. 將這兩個檔案上傳至 GitHub Pages (或其他靜態網頁代管服務) 發布。
4. 取得這兩個檔案發布後的 **公開網址**。

### 第四步：完成 LIFF 綁定與系統開機初始化
1. 回到 LINE Developers Console 的 **LIFF 頁籤**。
2. 將第三步取得的公開網址，分別對應填入兩個 LIFF 的 **Endpoint URL** 欄位中並儲存。
3. **系統自動建表：** 用手機或電腦的 LINE 開啟「老闆端 LIFF」連結（`https://liff.line.me/你的老闆端LIFF-ID`）。後端會自動在試算表中建立所需的資料表。
4. **設定管理員：** 打開試算表，切換到「管理員名單」工作表，在「LINE UserId」欄位填入你自己的 LINE User ID。
5. 重新開啟老闆端 LIFF 即可開始營運設定！

### 第五步：設定 LINE 官方帳號圖文選單 (打造專屬點餐入口)
1. 前往 **[LINE 官方帳號管理後台](https://manager.line.biz/)** 並登入你的店鋪官方帳號。
2. 進入 **主頁 > 圖文選單**，點擊右上角的「建立」。
3. 設定標題、顯示期間與上傳設計好的點餐按鈕圖片。
4. 在「動作設定」中，類型選擇 **連結 (URL)**。
5. 填入你的 **顧客端 LIFF URL**（格式：`https://liff.line.me/你的顧客端LIFF-ID`）。
6. 點擊「儲存」。顧客進入聊天室時點擊選單，即可無縫開啟點餐系統！
